[8/27/2025 7:58 PM] rumbledethumps
The rules are simple. You always need to align. 8 bits access needs 8 bit alignment, 16 bit needs 16 bits, 32 bits needs 32 bits. So when you say "unaligned accesses", there is no such thing.


[8/27/2025 8:14 PM] markrvm

REGSW(0xFFF1) is split into two accesses automatically. This is a very complicated subject you're trying to understand. The AHB can allow for some unaligned access, and the M33 can do things too. I think the rp2040 AHB allowed access to any 16 bits inside of a  32 bit word but couldn't cross 32 bit boundaries, which is what most of this code was written for. ```c
void opl_write(uint8_t reg, uint8_t data) {
    RIA.addr0 = 0xFF00;
    RIA.step0 = 1;
    
    RIA.rw0 = reg;   // Write Index
    
    // Short delay
    __asm__("nop"); 
    __asm__("nop");

    RIA.rw0 = data;  // Write Data
    
    // Longer delay: approx 24 microseconds
    // This ensures the OPL2 is ready for the NEXT command
    for(uint8_t i=0; i<32; i++) { __asm__("nop"); }
}```

```#include <rp6502.h>
#include <stdint.h>
#include <stdio.h>
#include "opl.h"
#include "instruments.h"

// --- Hardware Configuration ---
#ifdef USE_NATIVE_OPL2
    #define OPL_ADDR 0xFE00
#else
    #define OPL_ADDR 0xFF00
#endif

// C Major 7 Chord (MIDI 60, 64, 67, 71)
const uint8_t chord_notes[4] = {60, 64, 67, 71};

// A simple Brass/String-like patch
const OPL_Patch test_patch = {
    0x01, 0x00, 0xF1, 0x55, 0x00, // Modulator
    0x01, 0x00, 0xF2, 0x55, 0x00, // Carrier
    0x06                          // Feedback
};

// --- Main Test Loop ---

int main(void) {
    uint8_t v_last;
    uint8_t frame_counter = 0;

    // 1. Hardware Initialization
    OPL_Config(1, OPL_ADDR);
    OPL_Init();

    v_last = RIA.vsync;

    while (1) {
        while (RIA.vsync == v_last);
        v_last = RIA.vsync;

        // Chord trigger.. adjust as neded.
        if (frame_counter++ == 6) {
            frame_counter = 0;
            printf("Retriggering C-Maj-7 Chord...\n");

            for (uint8_t i = 0; i < 4; i++) {
                OPL_NoteOff(i);
                OPL_SetPatch(i, &test_patch);
                OPL_NoteOn(i, chord_notes[i]);
            }
        }
    }
    return 0;
}```

[2/28/2026 10:34 PM] rumbledethumps
Windows boots with num lock on, Linux off. Linux is wrong. The designer of that keyboard decided they would ignore how HID keyboards are supposed to work and make it Linux-only. A layer of configuration just adds more fucked up UX to an already fucked up UX so that's not a good solution. This is a rare circumstance because very few companies can get away with making keyboards that eschew Windows.

This is new to 0.18 because TinyUSB regressed some time ago and wasn't sending the LEDs at boot. Num-lock was always on because that's how it's been since the IBM AT introduced dedicated cursor keys in 1994, seven years before Linux.

Since it's a RPi branded keyboard we should probably have a PID/VID exception.

[4/10/2026 9:13 AM] voidas_pl
The result is predictable and explains several important things:

Why CPU won (1.37s vs 2.22s)? (3dqubePRE.c 3dcubePREMC.c examples)

The coprocessor has a fixed overhead per call — parameter setup in RIA, writing the opcode to $FFEF, spin-wait on $FFF1, reading the result. Count how many such calls precompute_frame makes per vertex:

mth_itof × 4, mth_mulf × 8, mth_addf × 2, mth_subf × 3, mth_sinf, mth_cosf, mth_divf, mth_ftoi × 3
That is ~23 RIA calls per vertex, ~189 per frame → ~34 000 RIA round-trips in total (180 frames × 189 calls).

The integer version instead:

sin/cos → zero computation — plain table lookups
multiplications → cc65 lmul — slow, but local (registers + zero page only, no bus traffic)
The earlier result (4× faster with coprocessor) is not contradicted — that benchmark compared sin via the Bhaskara formula (2× 32-bit lmul + ldiv) against the coprocessor. Here sin is a table — cost = a few branches and a memory read.

Conclusion:

Scenario |    Coprocessor |    Integer 65C02
sin/cos/sqrt/exp |    wins — 6502 has no alternative |    not feasible directly
multiply/divide + many calls | loses (RIA overhead) | wins (local)
few calls, complex operations | wins | slow
The coprocessor makes sense when the computation cost is large relative to the RIA round-trip cost — that is, for transcendental functions and float32, not for integer multiplications that cc65 can handle on its own.

Anyway - solution works.
